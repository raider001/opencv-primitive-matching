# OpenCV Pattern Matching — Testing Framework Plan

## Overview

This project tests various OpenCV image matching and shape detection techniques against a library
of synthetically generated reference images (lines, circles, grids, text, etc.) placed into a
large catalogue of scene images. Scenes include clean placements, scaled/rotated variants, and
**negative scenes** (no reference present at all) to measure false-positive rates.

All images are generated programmatically at test-time by `SceneLibraryGenerator` — no external
image files are committed to the repository. Running `mvn test` regenerates everything fresh.

---

## Reference Image Library

Reference images are simple, synthetic graphics generated by `ReferenceImageFactory`.
Each is produced at a standard **128 × 128 px** canvas in **full BGR colour** — not greyscale.

### Colour Model

Every reference image has three colour properties assigned by `ReferenceImageFactory`:

| Property | Description |
|----------|-------------|
| **Foreground colour** | The colour of the shape itself (stroke or fill) |
| **Background fill** | The canvas background — solid colour, gradient, or noise |
| **Gradient option** | Some references use a linear or radial gradient painted onto the shape itself |

#### Foreground colour palette (cycling across references)
Rather than always using white, each reference ID is assigned a foreground colour from this
fixed palette, cycling in order so the full palette is exercised:

| Slot | Colour | BGR value |
|------|--------|-----------|
| 0 | White | `(255, 255, 255)` |
| 1 | Red | `(0, 0, 220)` |
| 2 | Green | `(0, 200, 0)` |
| 3 | Blue | `(220, 0, 0)` |
| 4 | Yellow | `(0, 220, 220)` |
| 5 | Cyan | `(220, 220, 0)` |
| 6 | Magenta | `(220, 0, 220)` |
| 7 | Orange | `(0, 140, 255)` |

#### Background fill variants (per reference)
Each reference is rendered on one of four background fills, selected by
`referenceIndex % 4`:

| Slot | Background |
|------|------------|
| 0 | Solid black `(0, 0, 0)` |
| 1 | Solid dark grey `(40, 40, 40)` |
| 2 | Horizontal colour gradient (dark blue → dark red) |
| 3 | Radial gradient (dark centre → mid-grey edge) |

> ⚠️ **Analysis note — gradient reference backgrounds:**
> Slots 2 and 3 use gradient-filled canvases behind the shape. This is intentional (we want
> to observe how each technique handles non-uniform backgrounds), but it is expected to
> introduce complications during analysis:
>
> - **Template Matching** — the gradient becomes part of the template's pixel signature.
>   If the scene background does not share that gradient, even a perfect shape placement will
>   score lower than expected. This will likely produce misleadingly low scores on Cat A/B
>   scenes where the scene background differs from the reference canvas background.
>
> - **Histogram Comparison** — the gradient shifts the colour distribution of the reference,
>   making it harder for the matcher to isolate the foreground shape's colour contribution.
>
> - **Colour Pre-Filter (`_CF` variants)** — the gradient background will partially survive
>   the HSV threshold if its hue overlaps with the foreground colour, potentially widening
>   the effective mask and reducing precision.
>
> - **Feature Matching / Hu Moments / Hough** — less affected, since these operate on edges
>   or keypoints rather than raw pixel values, but the gradient adds edge noise at the
>   canvas border that may generate spurious keypoints or contours.
>
> **The reference image backgrounds are deliberately kept as-is** so that results reflect the
> real added complexity. When interpreting low scores on gradient-background references,
> check whether the drop is caused by the shape being hard to detect or simply by the
> background mismatch between the reference canvas and the scene canvas.
> The HTML report should flag gradient-background references with a ⚠️ indicator in the
> reference thumbnail column of the summary table.

#### Gradient shape fill variants
For every **filled** shape (e.g. `CIRCLE_FILLED`, `RECT_FILLED`, `TRIANGLE_FILLED`), an
additional gradient-fill variant is generated where the shape interior is painted with a
linear gradient from the foreground colour to a lighter/complementary tone, rather than a
flat fill. This is handled entirely inside `ReferenceImageFactory` using `Imgproc.line`
scanline painting over the filled mask.

## Scene Complexity Model

Every scene — regardless of category — is rendered on top of a **background layer** chosen from
the background pool below. This ensures that even "clean" positive scenes are not trivially easy.
Backgrounds are full **BGR colour** — not greyscale — so colour-sensitive techniques (histogram,
feature matching) are tested against realistically coloured scenes.

### Background Pool (used across all categories)

Backgrounds are grouped into four complexity tiers. Colour is present at every tier above Minimal.

#### Tier 1 — Minimal (baseline only)
| ID | Description | Colour |
|----|-------------|--------|
| `BG_SOLID_BLACK` | Plain black | None |
| `BG_SOLID_WHITE` | Plain white | None |
| `BG_SOLID_GREY` | 50% grey | None |

#### Tier 2 — Low complexity (colour gradients)
| ID | Description | Colour |
|----|-------------|--------|
| `BG_GRADIENT_H_GREY` | Horizontal greyscale gradient (black→white) | Greyscale |
| `BG_GRADIENT_V_GREY` | Vertical greyscale gradient (black→white) | Greyscale |
| `BG_GRADIENT_H_COLOUR` | Horizontal colour gradient (deep blue→deep red) | Blue→Red |
| `BG_GRADIENT_V_COLOUR` | Vertical colour gradient (dark green→dark purple) | Green→Purple |
| `BG_GRADIENT_RADIAL_GREY` | Radial gradient — bright centre, dark edges | Greyscale |
| `BG_GRADIENT_RADIAL_COLOUR` | Radial gradient — warm centre (orange), cool edges (teal) | Orange→Teal |
| `BG_NOISE_LIGHT` | Low-level Gaussian noise (σ=8) on a mid-grey base | Near-grey |

#### Tier 3 — Medium complexity (structured colour)
| ID | Description | Colour |
|----|-------------|--------|
| `BG_NOISE_HEAVY` | Heavy Gaussian noise (σ=35) on a mid-grey base | Multi |
| `BG_GRID_FINE` | Dense fine grid lines (every 16px) in dark cyan on black | Cyan on black |
| `BG_GRID_COARSE` | Coarse grid lines (every 64px) in dark blue on dark grey | Blue on grey |
| `BG_HATCHING` | Diagonal hatching at 45° in dark green on dark grey | Green on grey |
| `BG_RANDOM_LINES` | 20–40 randomly placed/angled line segments in random colours | Multi |
| `BG_RANDOM_CIRCLES` | 10–20 randomly placed circle outlines in random colours | Multi |

#### Tier 4 — High complexity (rich colour)
| ID | Description | Colour |
|----|-------------|--------|
| `BG_RANDOM_MIXED` | 30–50 random lines, circles, rectangles — each a random colour | Multi |
| `BG_DENSE_TEXT` | Tiled random alphanumeric chars in random colours on dark bg | Multi |
| `BG_CIRCUIT_LIKE` | PCB-style orthogonal traces in green/gold on dark brown | Green/Gold |
| `BG_ORGANIC` | Irregular blobs + curved lines painted in warm/cool colour bands | Multi |
| `BG_COLOURED_NOISE` | Full-frame RGB noise (each channel independently randomised) | Full colour |

**22 backgrounds total** — each scene is assigned one background deterministically by
`(referenceIndex + variantIndex) % 22` so the full pool is exercised evenly.

> **Why colour matters for testing:**
> - **Template Matching** — operates on raw pixel values, so colour mismatches directly reduce score
> - **Feature Matching** — typically converts to greyscale internally; colour backgrounds test robustness
> - **Histogram Comparison** — directly measures colour distribution; coloured backgrounds are maximum challenge
> - **Hough Transforms** — works on edge maps after greyscale conversion; colour is largely transparent
> - **Morphology / Hu Moments** — thresholds to binary first; colour of background affects threshold quality

## Scene Catalogue

Scenes are generated by `SceneGenerator` and stored in a `SceneCatalogue` object (in-memory `Mat`
list). Each scene is 640 × 480 px in **full BGR colour**. Scenes are grouped into four categories:

### Category A — Clean Placement (reference present, no transform)
The reference is placed at the centre of the scene over a background from the pool.
No scaling, rotation, noise, or degradation applied to the reference itself.
Four backgrounds are used per reference — one from each complexity tier:
`BG_SOLID_BLACK` · `BG_GRADIENT_H_COLOUR` · `BG_NOISE_LIGHT` · `BG_RANDOM_MIXED`

- **47 references × 4 backgrounds = 188 scenes**

The reference is transformed before placement. Each variant is paired with a mid-to-high
complexity colour background (`BG_RANDOM_MIXED`, `BG_CIRCUIT_LIKE`, or `BG_GRADIENT_RADIAL_COLOUR`)
cycling by `variantIndex % 3`.

## Test Execution Model

### One test method per technique

Each test class contains **exactly one `@Test` method**. Inside it, a double loop drives the full
run. `@BeforeAll` handles library loading and catalogue construction. `@AfterAll` triggers output
generation, including the HTML report for that technique.

```
@BeforeAll
  → nu.pattern.OpenCV.loadShared()
  → SceneCatalogue.build()              // generates all 1,267 scenes in-memory
  → create test_output/<technique>/ directories

@Test  analyseAllScenesAgainstAllReferences()
  for each REFERENCE IMAGE (47):
      for each SCENE ENTRY in CATALOGUE (1,267):
          result = matcher.run(scene.mat, reference.mat)
          // result = { methodVariant, scorePercent, boundingRect, elapsedMs, annotatedMat }
          collect result → results list

@AfterAll
  → AnalysisOutputWriter.saveAnnotatedImages(results)   // per-scene PNGs with bbox + score
  → AnalysisOutputWriter.saveReferenceGrids(results)    // per-reference side-by-side grid PNGs
  → AnalysisOutputWriter.printAsciiTable(results)       // console summary table
  → HtmlReportWriter.write(results, "test_output/<technique>/report.html")
```

### HTML Report structure (per technique)

The HTML report is a self-contained single file with embedded base64 images so it can be opened
directly from the filesystem without a server. It is generated by `HtmlReportWriter` and contains:

1. **Header** — technique name, total scenes run, total references, date/time, total elapsed ms.

2. **Summary table** — one row per reference image, columns:
   - Reference ID & thumbnail
   - Avg score % across all Cat A scenes (clean)
   - Avg score % across all Cat B scenes (transformed)
   - Avg score % across all Cat C scenes (degraded)
   - False-positive rate across all Cat D scenes (negative)
   - Min / Max / Avg elapsed ms

3. **Per-reference accordion sections** — expandable, one per reference (41 total). Each contains:
   - A labelled grid of all scene results (scene thumbnail + bbox overlay + score % label)
   - Colour-coded score cells: 🟢 ≥80% · 🟡 50–79% · 🔴 <50% (or inverted for distance metrics)
   - A breakdown table: scene category · variant label · background ID · score % · location · ms

4. **Cross-technique comparison panel** (in `MatchingBenchmarkTest` report only) — a table with
   one row per scene variant and one column per technique, allowing direct side-by-side comparison
   of every method's score on the same input.

5. **Footer** — link to `benchmark_summary.csv` for download.

### Final benchmark report

`MatchingBenchmarkTest` is an additional test class that imports the saved `List<AnalysisResult>`
from each technique (written to a shared `results/` cache by each test's `@AfterAll`), then
produces a single unified `test_output/benchmark/report.html` and `benchmark_summary.csv`.

### Expected output volume per technique:
- **~59,549 annotated images** (47 references × 1,267 scenes)
- **47 reference grid PNGs**
- **1 technique HTML report** (`test_output/<technique>/report.html`)
- **1 ASCII table** printed to console

### Total across all 9 techniques + benchmark:
- ~535,941 annotated images (note: consider a `--slim` flag to skip saving all PNGs and only
  write the HTML report with embedded thumbnails, keeping output manageable)
- 9 technique HTML reports
- 1 unified benchmark HTML report + `benchmark_summary.csv`

## Performance Metrics

### What is measured

Every `AnalysisResult` already captures `elapsedMs` for the actual match call. The performance
layer extends this with a `PerformanceProfile` that is computed by `PerformanceProfiler` during
`@AfterAll` — after all results are collected — so it does not interfere with timing accuracy.

#### Per-result timing fields (collected during the double loop)
| Field | Description |
|-------|-------------|
| `elapsedMs` | Wall-clock time for a single `matcher.run()` call on a 640×480 scene |
| `methodVariant` | Which sub-method was used (e.g. `TM_CCOEFF_NORMED`, `SIFT`) |
| `scenePx` | Total pixel count of the scene (width × height) |

#### Aggregated statistics (computed in `@AfterAll`)
| Statistic | Description |
|-----------|-------------|
| Min / Max / Avg / P95 ms | Per method variant across all 1,105 scenes |
| ms per megapixel | Normalised throughput: `elapsedMs / (scenePx / 1_000_000)` |
| Projected ms at resolution | Extrapolated from ms/MP (see table below) |
| Estimated heap MB at resolution | Based on `Mat` byte size formula (see below) |

---

### Resolution Extrapolation Model

All scenes in the test run are **640×480 (0.31 MP)**. Projections to higher resolutions are
calculated by `PerformanceProfiler.project(elapsedMs, sourceMp, targetMp)` using a
**linear scaling model** — i.e. time scales proportionally with pixel count. This is a
conservative estimate; in practice some algorithms (e.g. feature matching) scale sub-linearly
due to keypoint limits, while others (e.g. sliding-window template matching) can scale worse
than linear. Both a linear and a quadratic projection are shown.

| Resolution | Dimensions | Megapixels | Scale factor vs 640×480 |
|------------|------------|------------|------------------------|
| Test base  | 640 × 480  | 0.31 MP    | 1.0×  |
| 720p       | 1280 × 720 | 0.92 MP    | 3.0×  |
| 1080p      | 1920 × 1080| 2.07 MP    | 6.7×  |
| 1440p      | 2560 × 1440| 3.69 MP    | 11.9× |
| 4K UHD     | 3840 × 2160| 8.29 MP    | 26.8× |

#### Projected timing formula
```
linear_ms    = measured_ms × (targetMP / sourceMP)
quadratic_ms = measured_ms × (targetMP / sourceMP)²
reported_ms  = (linear_ms + quadratic_ms) / 2   // midpoint shown in report; both shown in CSV
```

### Memory Estimation Model

A raw OpenCV `Mat` (3-channel BGR, 8-bit) occupies exactly `width × height × 3` bytes.
`PerformanceProfiler.estimateHeapMb(width, height)` computes the working memory needed for
a single match operation, accounting for the intermediate `Mat` objects each algorithm creates:

| Algorithm | Intermediate Mats | Memory multiplier |
|-----------|-------------------|-------------------|
| Template Matching | result map (`W-w+1` × `H-h+1` × float32) | ~1.3× scene |
| Feature Matching | keypoint descriptor matrices (float32 or uint8) | ~2–4× scene |
| Hu Moments | sliding window crop per step | ~1.1× scene |
| Hough Transforms | edge map (grey, float) | ~1.5× scene |
| Generalized Hough | R-table + accumulator | ~3–5× scene |
| Histogram Comparison | per-window histogram arrays | ~1.2× scene |
| Phase Correlation | two complex (float64) FFT mats | ~4× scene |
| Morphology Analysis | binary threshold + contour storage | ~1.2× scene |
| Pixel Diff | diff mat (same size as scene) | ~2× scene |
| SSIM | Gaussian blur mats (6× per window, float32) | ~2.5× scene |
| Chamfer Distance | edge map + distance-transform float32 mat | ~1.8× scene |
| Fourier Shape | contour point arrays + two float32 DFT mats | ~1.2× scene |

#### Estimated heap per resolution (scene Mat only, before multiplier)

| Resolution | Raw scene Mat (3ch BGR 8-bit) | ×4 working estimate |
|------------|-------------------------------|---------------------|
| 640×480    | 0.88 MB  | 3.5 MB  |
| 720p       | 2.64 MB  | 10.6 MB |
| 1080p      | 5.93 MB  | 23.7 MB |
| 1440p      | 10.55 MB | 42.2 MB |
| 4K UHD     | 23.7 MB  | 94.9 MB |

> Note: These are **per-frame** estimates for a single match operation. Batch processing or
> holding the full scene catalogue in memory multiplies these figures by the number of concurrent
> frames held. The JVM heap overhead (~100–200 MB baseline) is not included.

### Performance Section in the HTML Report

The `HtmlReportWriter` includes a dedicated **Performance** tab in each technique report and in
the unified benchmark report, containing:

1. **Timing summary table** — one row per method variant, columns:
   - Min ms · Max ms · Avg ms · P95 ms (measured at 640×480)
   - Projected avg ms at: 720p · 1080p · 1440p · 4K (linear and quadratic)
   - ms/MP normalised throughput

2. **Resolution scaling chart** — an HTML/CSS bar chart (no JS library required) showing
   projected avg ms per variant at each resolution, grouped by variant, making it easy to
   visually compare how each algorithm scales.

3. **Memory estimation table** — one row per resolution, columns:
   - Scene Mat MB · Algorithm working MB (using per-algorithm multiplier) · Total estimated MB

4. **Interpretation notes** — auto-generated text per variant:
   - e.g. _"TM_CCOEFF_NORMED averages 4.2 ms at 640×480. At 4K this projects to ~112 ms
     (linear) — ~6.7 fps sustained, suitable for near-real-time use. At 1080p: ~28 ms (~36 fps)."_
   - e.g. _"SIFT averages 38 ms at 640×480. At 4K this projects to ~1,018 ms (linear) —
     not suitable for real-time use at this resolution without hardware acceleration."_

5. **Cross-technique performance comparison** (benchmark report only) — a single table showing
   avg ms + 4K projection for every technique side-by-side.

### `PerformanceProfiler` class

Added to `src/main/java/org/example/PerformanceProfiler.java`:

```
static PerformanceProfile profile(List<AnalysisResult> results, String methodVariant)
  → aggregates min/max/avg/p95 elapsedMs
  → computes ms/MP
  → projects to 720p, 1080p, 1440p, 4K (linear + quadratic)
  → estimates heap MB per resolution using per-algorithm multiplier map

static String interpretationNote(PerformanceProfile profile)
  → returns a human-readable sentence for the HTML report
```

`PerformanceProfile` is a record:
```
{ String methodVariant, long minMs, long maxMs, double avgMs, long p95Ms, double msPerMp,
  Map<String, double[]> projectedMs,   // key = "720p"/"1080p"/"1440p"/"4K", value = [linear, quadratic]
  Map<String, Double> estimatedHeapMb  // key = resolution label
}
```

---



Slides a template over a source image and computes a pixel-level similarity score at every position.

| Method Constant      | Description                              | Better When |
|----------------------|------------------------------------------|-------------|
| `TM_SQDIFF`          | Sum of squared pixel differences         | Lower score |
| `TM_SQDIFF_NORMED`   | Normalized SQDIFF (0–1)                  | Lower score |
| `TM_CCORR`           | Cross-correlation                        | Higher score |
| `TM_CCORR_NORMED`    | Normalized cross-correlation             | Higher score |
| `TM_CCOEFF`          | Correlation coefficient (mean-subtracted)| Higher score |
| `TM_CCOEFF_NORMED`   | Normalized CCOEFF (**most robust**)      | 1.0 = perfect |

- ✅ Fast, simple, exact pixel comparison — great for rigid unchanged templates
- ❌ No rotation or scale invariance; template must match the target region size exactly

---

### 2. Feature Matching (Keypoint Descriptors)

Detects distinctive keypoints in both images and matches their descriptors.

| Detector  | Notes                                                        |
|-----------|--------------------------------------------------------------|
| **SIFT**  | Highly accurate, scale & rotation invariant; patent-free since 2020 |
| **ORB**   | Fast binary descriptor, free alternative to SIFT             |
| **AKAZE** | Robust, patent-free, good for non-linear scale spaces        |
| **BRISK** | Fast binary, good for real-time use                          |
| **KAZE**  | More accurate than AKAZE but slower                          |

**Matching strategies:**
- `BFMatcher` (Brute Force) — exhaustive; uses `NORM_L2` (float) or `NORM_HAMMING` (binary)
- `FlannBasedMatcher` — approximate nearest-neighbor, faster for large descriptor sets
- Lowe's ratio test (threshold ~0.75) to filter good matches
- `findHomography` + RANSAC to verify geometric consistency

- ✅ Scale, rotation, and partial occlusion invariant
- ❌ Overkill for simple shapes; ORB can struggle on low-texture shapes; slower than template matching

---

### 3. Contour Shape Matching (Hu Moments)

Extracts shape contours, computes 7 Hu Moments (invariant to rotation, scale, and translation),
then compares shapes numerically.

**Pipeline:**
1. `Imgproc.findContours` — extract shape boundaries
2. `Imgproc.moments` — compute raw moments
3. `Imgproc.HuMoments` — derive 7 invariant descriptors
4. `Imgproc.matchShapes(contour1, contour2, method, 0)` — returns `0.0` for identical shapes

**Comparison methods:** `CONTOURS_MATCH_I1`, `CONTOURS_MATCH_I2`, `CONTOURS_MATCH_I3`

- ✅ Rotation, scale, and translation invariant — well suited for classifying primitive shapes
- ❌ Sensitive to contour noise; only compares shape silhouette, ignores internal texture

---

### 4. Hough Transforms

Detects parametric geometric primitives directly in edge maps.

| Method            | Detects                    | API                       |
|-------------------|----------------------------|---------------------------|
| `HoughLines`      | Infinite lines (ρ, θ)      | `Imgproc.HoughLines`      |
| `HoughLinesP`     | Line segments (probabilistic, faster) | `Imgproc.HoughLinesP` |
| `HoughCircles`    | Circles (center + radius)  | `Imgproc.HoughCircles`    |

- ✅ Direct detection of lines and circles; robust to gaps and noise; no template needed
- ❌ Only handles specific primitives natively (no triangles/rectangles); requires parameter tuning

---

### 5. Generalized Hough Transform

Extends Hough to arbitrary template shapes — not just lines and circles.

- `GeneralizedHoughBallard` — detects translated instances of a template shape
- `GeneralizedHoughGuil` — additionally handles rotation and scale

- ✅ Handles arbitrary shapes with rotation and scale variance; template-driven
- ❌ Slow; high memory usage; harder to tune

### 6. Histogram Comparison (`Imgproc.compareHist`)

Computes intensity/colour distribution histograms for two images and compares them.

| Method                  | Range    | Better When |
|-------------------------|----------|-------------|
| `HISTCMP_CORREL`        | -1 to 1  | Higher      |
| `HISTCMP_CHISQR`        | 0 to ∞   | Lower       |
| `HISTCMP_INTERSECT`     | 0 to sum | Higher      |
| `HISTCMP_BHATTACHARYYA` | 0 to 1   | Lower       |

- ✅ Fast; spatially independent; works on colour distributions
- ❌ Loses all spatial information — two different shapes with the same colour distribution score identically

---

### 7. Phase Correlation (`Video.phaseCorrelate`)

FFT-based cross-correlation that computes the sub-pixel **translation offset** between two same-size images.
An optional Hanning window (`createHanningWindow`) reduces boundary artifacts.

- ✅ Sub-pixel accuracy; very fast (FFT-based); ideal for detecting pure translation
- ❌ Translation only — no rotation or scale handling; both images must be the same size

### 8. Morphological Shape Analysis

Classifies shapes by geometry after binarisation — no template or descriptor needed.

| Tool                      | Purpose                                               |
|---------------------------|-------------------------------------------------------|
| `approxPolyDP`            | Approximates contour to polygon (3=triangle, 4=rect, many=circle) |
| `isContourConvex`         | Convexity check                                       |
| `minEnclosingCircle`      | Fits a bounding circle                                |
| `minAreaRect`             | Fits a minimum bounding rectangle                     |
| Circularity ratio         | `4π × area / perimeter²` → 1.0 = perfect circle      |
| Aspect ratio / solidity   | Additional shape descriptors                          |

- ✅ Extremely lightweight; deterministic; perfect for classifying known primitive shapes
- ❌ Requires clean binarisation; does not generalise beyond simple shapes

---

### 9. Absolute Difference / Pixel Diff

`Core.absdiff()` + `Core.sumElems()` — computes a simple pixel-level dissimilarity score.
Used as a fast baseline comparison metric.

### 11. SSIM — Structural Similarity Index (Milestone 17)

Decomposes image similarity into three perceptually meaningful components using Gaussian-weighted
local statistics:

| Component   | Formula                                         | Captures                       |
|-------------|------------------------------------------------|--------------------------------|
| Luminance   | `(2μxμy + C1) / (μx² + μy² + C1)`             | Mean brightness match          |
| Contrast    | `(2σxσy + C2) / (σx² + σy² + C2)`             | Local variance match           |
| Structure   | `(σxy + C3) / (σxσy + C3)`                     | Normalised cross-correlation   |

Combined SSIM = luminance × contrast × structure ∈ [−1, 1]; mapped to 0–100%.
A sliding 128×128 window at stride 8 localises the best match within the scene.

- ✅ More tolerant of uniform brightness/contrast shifts than Pixel Diff or Template Matching
- ✅ The structure component captures edge alignment without requiring explicit edge detection
- ❌ Not scale- or rotation-invariant — positions it clearly as a perceptual complement to TM

---

### 12. Chamfer Distance Matching (Milestone 18)

Matches a shape by sampling its **edge pixels into a precomputed distance-transform field** of
the scene's edge map.

**Pipeline:**
1. Canny edge detection on the reference → list of edge pixel coordinates
2. Canny edge detection on the scene → `Imgproc.distanceTransform` (L1 or L2) → distance field
3. Slide a 128×128 window; for each position, look up every reference edge point in the field
4. Average sampled distance = Chamfer distance; score = `1 / (1 + avgDist) × 100`

| Distance type | Characteristic                              |
|---------------|---------------------------------------------|
| `DIST_L1`     | City-block; faster; more robust on occlusion |
| `DIST_L2`     | Euclidean; penalises outlier distances more  |

- ✅ Partial occlusion is handled gracefully — missing edge fragments contribute bounded distance
- ✅ Distance field precomputed once per scene; sampling is very fast
- ✅ CF variants suppress background edges before extraction, reducing false candidates
- ❌ Not fully rotation-invariant (edge layout changes with rotation)

---

### 13. Fourier Shape Descriptors (Milestone 19)

Represents a shape as a **1-D frequency-domain signature** derived from its boundary contour.

**Pipeline:**
1. Extract the largest external contour from a binarised image
2. Resample uniformly to 128 points along arc length
3. Encode as a complex signal: `z(t) = (x(t) − cx) + j(y(t) − cy)` (centroid-centred)
4. Apply 1-D DFT via `Core.dft` on a two-channel packed Mat (real + imag channels)
5. Compute magnitude spectrum; skip DC (k=0); normalise by coefficient at k=1 (scale invariance)
6. Keep first 32 normalised magnitudes as the descriptor
7. Compare with reference descriptor via L2 distance; score = `1 / (1 + dist) × 100`

**Why it is inherently invariant:**
- **Rotation** — rotating the contour by angle θ multiplies every coefficient by `e^(jkθ)`.
  This is a global phase shift that vanishes entirely in the **magnitude** spectrum.
- **Scale** — normalising by the k=1 coefficient (the "fundamental frequency") makes all
  other coefficients dimensionless ratios independent of absolute size.
- **Starting point** — the starting point of the contour traversal shifts all phases by a
  constant, which again vanishes in the magnitude spectrum.

- ✅ True rotation and scale invariance — no RANSAC, no homography, no moment tensor
- ✅ Low frequency coefficients capture gross shape; higher ones capture fine detail
- ✅ Naturally suited to the contour-rich synthetic reference library
- ❌ Requires a clean, connected contour — breaks on heavily occluded or cluttered shapes

### 10. Colour-First Region Proposal (CF1)

A **pipeline architecture** rather than a standalone technique. Instead of masking the full
scene and running a matcher over the entire masked frame, this approach uses the colour
threshold as a **spatial filter** that proposes candidate regions *before* the expensive
matcher is invoked.

#### Pipeline

```
Scene (640×480 BGR)
  │
  ▼
ColourPreFilter.apply()          ← HSV inRange → binary mask
  │
  ▼
findContours / connectedComponents ← candidate blobs
  │
  ▼
Merge overlapping rects           ← reduce to K distinct windows
  Pad each window to ≥ 128×128   ← ensure matcher has enough context
  Clamp to scene bounds
  │
  ▼
For each candidate window:
  Run base matcher (e.g. TM_CCOEFF_NORMED, CONTOURS_MATCH_I1)
  │
  ▼
Return best-scoring candidate bbox + score
```

#### Key characteristics

| Property | Value |
|----------|-------|
| Speed vs full-scene CF | **5–50× faster** — only K small windows searched (K typically 1–5) |
| Speed vs base (no CF) | **Faster on cluttered scenes**, similar on simple scenes |
| Score vs base CF | **Same or better** — same matcher on fewer, more targeted crops |
| D_NEGATIVE behaviour | Zero candidates → score = 0% automatically |
| White/grey references | Falls back gracefully — proposes nothing → full-scene fallback |

#### Key classes

- `ColourFirstLocator` — the proposal engine; generic, stateless, reusable by any matcher
- Each of the 9 existing matchers gains 2 extra CF1 variant paths internally (no new wrapper classes needed)

#### What to look for in comparison

| Scenario | CF (full-scene mask) | CF1 (region proposal) |
|----------|---------------------|----------------------|
| A_CLEAN — isolated shape | Similar speed, same score | Same score, possibly faster |
| B_TRANSFORMED — busy scene | Searches full masked scene | Proposes 1–3 candidates only |
| D_NEGATIVE — wrong colour | Low score but still searches everything | **Zero candidates → instant 0%** |
| C_DEGRADED — noisy colour | Mask degrades but still searches | Fewer, noisier candidates |
| Cluttered multi-colour scene | Mask has many surviving pixels | Splits into distinct blobs |

- ✅ Fastest approach for colour-distinctive targets on busy backgrounds
- ✅ D_NEGATIVE returns 0% with near-zero compute cost
- ❌ Requires target to have a distinctive foreground colour
- ❌ Degrades if target colour matches background (same limitation as all CF variants)

## Technique 10 — Colour Pre-Filter (variant extension for all 9 base techniques)

### Concept

Before running any of the 9 base techniques, a **colour isolation pre-processing step** is
applied to both the scene and the reference image. This converts the colour match into a
shape-only binary match problem by:

1. Converting the scene and reference from BGR → HSV colour space
2. Extracting the **dominant foreground colour** of the reference using `Core.minMaxLoc` on
   the Hue channel (or reading it directly from `ReferenceId` metadata)
3. Applying `Core.inRange` with a configurable HSV tolerance window to produce a **binary mask**:
   pixels within the colour range → white (255), everything else → black (0)
4. Running the base technique's matcher on the **masked binary images** instead of the original
   colour images

This is a **variant extension** — it wraps each of the 9 base matchers. Each pre-filtered
variant is identified by the suffix `_CF` (e.g. `TM_CCOEFF_NORMED_CF`, `SIFT_CF`, `ORB_CF`).

### Implementation

`src/main/java/org/example/ColourPreFilter.java`:

```
static Mat apply(Mat bgrImage, Scalar hsvLower, Scalar hsvUpper)
  → converts BGR → HSV via Imgproc.cvtColor
  → applies Core.inRange(hsv, lower, upper, mask)
  → returns the binary mask (CV_8UC1, 0 or 255)

static ColourRange extractReferenceColourRange(ReferenceId id, double hueTolerance)
  → looks up the known foreground BGR colour for this ReferenceId
  → converts to HSV
  → returns ColourRange { hsvLower, hsvUpper } with ±hueTolerance on Hue,
    ±40 on Saturation, ±40 on Value
```

`ColourRange` record: `{ Scalar hsvLower, Scalar hsvUpper, String label }`

The default `hueTolerance` is **15°** (out of 180° in OpenCV HSV). A tighter tolerance (e.g. 8°)
is also run as a second sub-variant, giving two CF variants per technique:
- `_CF_LOOSE` — ±15° hue tolerance
- `_CF_TIGHT` — ±8° hue tolerance

This doubles the CF variants to **18 colour-filtered variants** across all 9 techniques.

### HSV tolerance window visualised

```
Reference colour: Red (H≈0°/180°, S=220, V=220)
CF_LOOSE mask:    H ∈ [165°, 15°] (wraps), S ∈ [180, 255], V ∈ [180, 255]
CF_TIGHT mask:    H ∈ [172°, 8°]  (wraps), S ∈ [192, 255], V ∈ [192, 255]
```

### Why this is interesting to compare

| Scenario | Base technique result | CF variant result |
|----------|-----------------------|-------------------|
| Coloured shape on matching-colour background | Confused | Confused (both pass threshold) |
| Coloured shape on contrasting background | Good | Better — background noise eliminated |
| Shape present but colour-shifted (Cat C) | Good (colour-agnostic) | Worse — fails threshold |
| Negative scene with same-colour clutter | High FP risk | Reduced FP — colour gate filters clutter |
| Greyscale or near-grey reference (e.g. LINE_H white) | Baseline | Minimal benefit — white has low saturation |

### Performance impact of the pre-filter step

The colour pre-filter itself adds one `cvtColor` + one `inRange` call per image pair.
At 640×480 this is typically **< 1 ms** and at 4K typically **< 8 ms** — negligible compared
to the base technique cost. This overhead is tracked separately in `AnalysisResult` as
`preFilterElapsedMs` and reported in the Performance tab.

### Memory impact

The pre-filter produces one additional `Mat` per image (the binary mask, CV_8UC1).
Memory multiplier for CF variants = base multiplier + `(width × height × 1) / 1_048_576` MB.
For 4K this adds ~7.9 MB per mask.

## With and Without Colour Filter — Design Mandate

> **Every technique must be run in three modes: base, CF_LOOSE, and CF_TIGHT.**

This is a hard requirement, not an optional extension. The purpose of the project is to
**compare** matching strategies — and the colour pre-filter is a first-class comparison axis.

### What this means in practice

Each technique test class (Milestones 7–15) runs a **single double loop** over
88 references × 2,332 scenes. Inside that loop, for each (reference, scene) pair, the matcher
is invoked **three times**:

| Pass | Variant suffix | Input to matcher |
|------|---------------|-----------------|
| 1 | *(none)* — base | Full-colour scene + reference, no pre-processing |
| 2 | `_CF_LOOSE` | Scene and reference both passed through `ColourPreFilter.apply()` at ±15° hue |
| 3 | `_CF_TIGHT` | Scene and reference both passed through `ColourPreFilter.apply()` at ±8° hue |

All three `AnalysisResult` objects are appended to the same result list, so every downstream
consumer (ASCII table, grid images, HTML report) sees and compares all three together.

### What to look for in the comparison

| Scene condition | Expected base result | Expected CF result |
|-----------------|---------------------|--------------------|
| Cat A — clean colour match | Good | Better (noise eliminated) |
| Cat B — scaled / rotated | Good | Similar or better |
| Cat C — colour-shifted (hue +40°) | Good | Worse — fails colour gate |
| Cat C — noise / blur / occlusion | Good | Similar (mask degrades with noise) |
| Cat D — negative (no shape) | Low FP | Lower FP — colour gate rejects background |
| White reference (low saturation) | Baseline | Little benefit — white has no hue |
| Red/orange reference | Baseline | Needs hue wrap-around handling |

### HTML report structure (per technique)

Every technique report (`test_output/<technique>/report.html`) contains three tabs:

1. **Results** — average match score per reference per category, one column group per mode
   (base · CF_LOOSE · CF_TIGHT), colour-coded 🟢🟡🔴
2. **Base vs CF Comparison** — delta table showing `CF_LOOSE − base` and `CF_TIGHT − base`
   per reference per category; cells colour-coded 🟢 = CF improved · 🔴 = CF degraded
3. **Performance** — timing (base ms + pre-filter ms overhead), projection to higher resolutions,
   memory estimates — all three modes side-by-side

## Test Philosophy

> These are **analytical tests**, not pass/fail unit tests. The goal is to run every technique
> against the same inputs and produce visual + numeric output for human analysis and comparison.
> JUnit 5 is used purely as a structured runner — **no assertions, no failures**.

### Rules
- **One `@Test` method per test class.** The entire reference × scene matrix is driven by a
  double loop inside that single test method.
- **`@BeforeAll`** loads OpenCV and builds the `SceneCatalogue`.
- **`@AfterAll`** generates all output — annotated images, grid PNGs, ASCII console table,
  and the **self-contained HTML report** for that technique.
- Tests never throw, never assert, never fail. If a matcher errors on a scene, the result is
  recorded as score = 0% with an error flag, and execution continues.

### Per-test output
1. **Annotated PNG per scene** — scene image with coloured bounding box drawn at the best match
   location, labelled `"MethodVariant: 87.3%"`.
2. **Per-reference grid PNG** — all 1,267 scene results for one reference tiled side-by-side.
3. **ASCII results table** — printed to console on test completion.
4. **Self-contained HTML report** — `test_output/<technique>/report.html` with embedded base64
   thumbnails, colour-coded score table, expandable per-reference sections, and timing stats.

## Implementation Steps

1. **`ReferenceImageFactory`** — generates all 88 reference `Mat` images keyed by an `enum ReferenceId`.
   Uses `Imgproc.line`, `Imgproc.circle`, `Imgproc.rectangle`, `Imgproc.polylines`,
   `Imgproc.putText`, and `Imgproc.fillPoly` on 128×128 **BGR colour** canvases.
   - Foreground colours cycle through the 8-colour palette (white, red, green, blue, yellow,
     cyan, magenta, orange) by reference index
   - Canvas backgrounds cycle through 4 fills (solid black, solid dark grey, H colour gradient,
     radial gradient) by reference index
   - Gradient-fill variants use scanline painting over the filled mask for smooth linear colour fill
   - ⚠️ Gradient canvas backgrounds (slots 2 & 3) are kept intentionally — see the
     **Analysis note** in the Reference Image Library section for expected complications

2. **`BackgroundFactory`** — generates each of the 22 background `Mat` images at 640×480 in
   full BGR colour across 4 tiers (Minimal → Low → Medium → High complexity).

3. **`SceneGenerator`** — builds all four scene categories using `BackgroundFactory`:
   - Cat A: 47 refs × 4 colour backgrounds = 188 scenes
   - Cat B: 47 refs × 15 transform variants on colour backgrounds = 705 scenes
   - Cat C: 47 refs × 7 degradation variants (incl. colour shift) on high-complexity colour
     backgrounds = 329 scenes
   - Cat D: 45 negative scenes across all 22 backgrounds + coloured noise/shape variants

4. **`SceneCatalogue`** — calls `ReferenceImageFactory` + `BackgroundFactory` + `SceneGenerator`
   to build the full in-memory catalogue of **1,267** `SceneEntry` records, each holding:
   `{ ReferenceId, category, variantLabel, backgroundId, sceneMat }`.

5. **`AnalysisResult`** — shared result record:
   `{ methodName, referenceId, variantLabel, scorePercent, boundingRect, elapsedMs,
      preFilterElapsedMs, scenePx, annotatedMat, isError, errorMessage }`.
   `preFilterElapsedMs` is 0 for base variants and records the colour filter cost for CF variants.

6. **`ColourPreFilter`** — colour isolation pre-processing utility:
   - `apply(Mat bgrImage, Scalar hsvLower, Scalar hsvUpper)` → binary mask `Mat`
   - `extractReferenceColourRange(ReferenceId id, double hueTolerance)` → `ColourRange`
   - Produces `_CF_LOOSE` (±15° hue) and `_CF_TIGHT` (±8° hue) variants for every base technique
   - Each matcher class checks whether it is running in CF mode and calls `ColourPreFilter.apply()`
     on both scene and reference before executing the base algorithm

7. **`AnalysisOutputWriter`** — utility that:
   - Draws a coloured bounding box + `"MethodName: 87.3%"` label onto a copy of the scene
   - For CF variants, additionally overlays the binary mask as a semi-transparent tint
   - Assembles per-reference tiled grid images from a list of `AnalysisResult`
   - Saves annotated PNGs to `test_output/<technique>/<referenceId>/`
   - Prints a formatted ASCII results table and ASCII performance table to stdout

7. **`HtmlReportWriter`** — generates a self-contained `report.html` with embedded base64 images:
   - Summary table (one row per reference, columns per scene category + timing stats)
   - Colour-coded scores: 🟢 ≥80% · 🟡 50–79% · 🔴 <50%

## Notes

- **Native library loading:** openpnp bundles the native `.dll`/`.so` inside the JAR.
  Use `nu.pattern.OpenCV.loadShared()` in `@BeforeAll` — no manual `-Djava.library.path` needed.
- **SIFT availability:** openpnp 4.7.0-0 bundles `opencv_contrib`, so SIFT is available.
  SURF (`xfeatures2d`) may not be — skip with `Assumptions.assumeTrue` if unavailable.
- **In-memory images:** All test `Mat` objects are generated fresh per test run.
  No external image files are required.
- **Run tests:** `mvn test`
- **Run with coverage:** `mvn verify`
